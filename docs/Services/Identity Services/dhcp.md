# Overview

Even though DHCP doesn't directly support identity, it is too small to justify it's own subsystem. It also natively integrates with Microsoft AD because it can register itself as the domain's DHCP server and, in certain configurations, register DNS names on behalf of leases it issues.

## IN PROGRESS

This document summarizes the deployment of a Windows-based DHCP server in a segmented VLAN architecture, including relay integration, lease design, and troubleshooting lessons learned.

The focus is strictly on DHCP mechanics, relay behavior, and operational debugging—not identity tiering, firewall architecture, or broader infrastructure design.

# DHCP Server Deployment

The DHCP server was deployed as a domain-joined member server and authorized in Active Directory.

Core steps:
- Install DHCP role
- Authorize server in AD
- Create scopes
- Define lease duration
- Validate lease issuance on local VLAN

Scopes were configured using Ansible and PowerShell to maintain automation consistency.

# DHCP Relay

DHCP is broadcast-based at initialization.

A client without an IP sends:
`0.0.0.0:68 → 255.255.255.255:67`

Broadcast traffic does not cross L3 boundaries.

In segmented VLAN environments, relay is required to:
- Capture broadcast on client VLAN
- Convert to unicast
- Forward to DHCP server
- Relay server response back to client

## DHCP Relay Flow Explained

Unlike protocols like DNS, DHCP relay breaks connection symmetry.

There are three distinct traffic paths:

### A) Client → Relay

Broadcast:
src: 0.0.0.0:68
dst: 255.255.255.255:67

This hits the relay device locally.

### B) Relay → DHCP Server

Unicast:
src: <relay_vlan_ip>:67
dst: <dhcp_server_ip>:67

This is a new packet generated by the relay.

### C) DHCP Server → Relay → Client

Unicast back to relay:
src: <dhcp_server_ip>:67
dst: <relay_vlan_ip>:67

Relay then forwards:
src: <relay_vlan_ip>:67
dst: client:68

These are separate UDP conversations.
State tracking cannot unify them like symmetric protocols (e.g., DNS).

Explicit policy allowances are required for:
- UDP 67 inbound (client)
- UDP 67 outbound (to server)
- UDP 67 inbound (from server)
- UDP 68 outbound (to client)

## Relay Tool Selection

Several relay options were evaluated:

|Tool|Outcome|
|Legacy ISC relay|Rejected due to lifecycle concerns|
|Kea relay|Not cleanly packaged in target distro|
|Lightweight relay utilities|Limited packaging availability|
|dnsmasq|Selected|

dnsmasq was chosen because:
- Packaged and maintained in the target OS
- supports relay mode
- Lightweight footprint
- Reliable and well-documented
- Suitable for boundary routing devices

# Troubleshooting Events

This section details major events that occured during implementation, the associated root cause analysis, and the resolution mechanisms.

## Critical Implementation Detail: Interface Binding

A key discovery:

The interface= directive in dnsmasq does not only control listening behavior—it also influences packet handling and interface-level routing behavior.

Misunderstanding this can result in:
- Relay not activating
- Packets reaching the host but not being processed
- Silent failures

Correct interface scoping is essential.

## Major Troubleshooting Event: Hidden Firewall Controller
Symptoms:
- Client broadcast visible in packet capture
- No relay logs
- No upstream DHCP traffic
- No lease assignment

Root Cause:
- An OS-level firewall service (firewalld) was still active.
- It was inserting nftables rules ahead of custom rulesets, blocking DHCP traffic before it reached userspace.

Because multiple firewall controllers were present:
- Traffic appeared to reach the interface
- Custom rules did not appear to drop it
- Relay daemon never received packets

Resolution:
- The conflicting firewall service was fully disabled and masked.

Once removed, relay worked immediately.

Lesson:
- Only one firewall controller should manage packet filtering on a routing boundary device.
- Mixing abstraction layers introduces hidden policy and unpredictable behavior.

# Key Lessons Learned

DHCP relay breaks normal stateful assumptions

Broadcast-to-unicast translation must be explicitly allowed

Interface binding semantics matter more than expected

Multiple firewall controllers create invisible drop paths

Packet capture visibility does not guarantee userspace delivery
